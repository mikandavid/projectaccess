# Project Access Austria Website Guide

## Who We Are

Project Access Austria is the Austrian chapter of Project Access. The site serves students from Austria and South Tyrol who are interested in applying to selective international universities, especially through free mentoring, application resources, and the annual bootcamp.

The Austrian organization is legally represented as Projekt Hochschulzugang, an association based in Vienna. Keep the tone student-centered, encouraging, practical, and connected to the wider Project Access mission of making top universities more accessible regardless of background.

## Stack

- Static bilingual website for Project Access Austria.
- Source content lives in JSON files under `content/`.
- Page structure is defined with Jinja2 templates under `templates/`.
- `build.py` renders German pages at the repository root and English pages under `en/`.
- Styling is in the single global stylesheet `styles.css`.
- Runtime JavaScript is limited to the external Calendly badge loaded from `templates/base.html.j2`.
- Python dependencies: `Jinja2` (templating) and `Pillow` (WebP image conversion via `scripts/convert_assets_to_webp.py`), installed from `requirements.txt`.

## Build

Run:

```bash
python3 -m pip install -r requirements.txt
python3 build.py
```

Use `python3 build.py --clean` to remove generated HTML pages before rebuilding.

### Adding a new photo

Drop the original JPEG/PNG into `assets/`, then run:

```bash
python3 scripts/convert_assets_to_webp.py
```

This generates a `.webp` sibling for every raster file (skips ones already up to date). The site references the `.webp` version everywhere except the favicon and the JSON-LD `Organization.logo`, which stay as `pa-austria-logo.png` because Apple home-screen icons and Google's logo rich-result guidelines still require PNG/JPG. The script caps width at 2000 px (override with `--max-width`); pass `--force` to re-encode everything.

Originals stay in `assets/` as the editable masters. They are deployed to GitHub Pages too but no rendered HTML links to them, so they cost nothing at request time.

Generated HTML files include:

- German: `index.html`, `bootcamp.html`, `events.html`, `partners.html`, `team.html`
- English: `en/index.html`, `en/bootcamp.html`, `en/events.html`, `en/partners.html`, `en/team.html`

`build.py` also writes SEO/AI discovery files at the repository root:

- `sitemap.xml` (all pages with hreflang alternates)
- `robots.txt` (allow-all + sitemap reference)
- `llms.txt` (curated map for AI assistants per the llms.txt convention)

These three files are recreated on every build and removed by `--clean`.

The legacy URLs `mitmachen.html` and `en/get-involved.html` are kept as static `<meta http-equiv="refresh">` redirect stubs that point to the merged Team page. They are not generated by `build.py` and should not be deleted by `--clean`.

Prefer editing the JSON content, Jinja templates, or `styles.css`; rebuild generated HTML after source changes.

## Source Layout

- `build.py`: build pipeline, page map, language handling, localized anchors, template helpers, output paths, and SEO file generation (sitemap, robots, llms.txt).
- `content/_site.json`: shared site chrome, navigation, footer, language switch labels, Calendly badge settings, press links, plus SEO defaults (`site_url`, `default_og_image`, `theme_color`, favicon, full `organization` profile used by JSON-LD).
- `content/<page>.json`: bilingual page content for each logical page. Each page's `meta` block holds `title`, `description`, optional `keywords`, and an optional `og_image` override.
- `templates/base.html.j2`: shared HTML shell, full SEO/OG/Twitter metadata, hreflang/canonical links, font/css includes, header/footer includes, and Calendly script. Exposes `{% block og_type %}` and `{% block structured_data %}` for per-page schema.
- `templates/_partials/`: reusable template fragments such as header, footer, press section, and `structured_data.html.j2` (sitewide NGO/WebSite/WebPage/BreadcrumbList JSON-LD).
- `templates/<page>.html.j2`: page-specific layouts; bootcamp adds EducationEvent + FAQPage JSON-LD, events adds an EducationEvent for the next bootcamp, index adds an OfferCatalog of programmes.
- `styles.css`: global visual system, layout, responsive rules, and components.
- `CNAME`: custom domain for the live deployment.
- `.nojekyll`: keeps the static deployment compatible with GitHub Pages-style hosting.
- `docs/`: planning and source-content notes. Treat these as reference material, not generated output.

## Website Layout

The site is bilingual with German as the root-language output and English in `en/`.

Primary pages:

- Home: overview of mission, programme, bootcamp, engagement paths, team preview, events, and contact.
- Bootcamp: details for the application bootcamp, eligibility, activities, and calls to action.
- Events: events/news and related press context.
- Partners: sponsor and partner information.
- Team & join us: full team presentation grouped by area, plus mentoring/volunteer recruitment and CTAs.

Shared navigation is generated from `content/_site.json`. The header includes a CSS-only responsive menu toggle, language links, and optional page-level CTA from content.

## Design Notes

- Visual direction: warm cream/peach backgrounds, dark ink text, orange/yellow/coral/green accents, rounded cards, pill labels, gentle shadows, and human photography.
- Typography uses Inter from Google Fonts via `templates/base.html.j2`.
- Layout should stay responsive with fluid type, mobile-first grids, large tap targets, and images using fixed aspect ratios plus `object-fit: cover`.
- Avoid adding heavy animation or new frontend dependencies unless explicitly requested.

## Editing Guidance

- Keep content bilingual. Most user-facing strings should be represented in both `de` and `en` fields.
- If adding a page, update `PAGES` in `build.py`, add `content/<page>.json`, add `templates/<page>.html.j2`, and add navigation in `content/_site.json` if it belongs in the header.
- Use `page_href(...)`, `asset(...)`, `anchor(...)`, and `t(...)` helpers in templates instead of hardcoding cross-language URLs.
- Keep generated HTML consistent by rebuilding after source edits.
- Do not edit `.venv/` or generated dependency files.
